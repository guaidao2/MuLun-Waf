# Mulun-WAF：基于侧枝决策架构的轻量智能 Web 应用防火墙

**玄幕安全团队 — guaidao2**

2026 年 7 月

---

## 摘要

本文提出 Mulun-WAF，一种基于侧枝决策架构（Sidecar Decision Architecture）的轻量智能 Web 应用防火墙。与传统依赖正则规则或大型语言模型的 WAF 方案不同，Mulun-WAF 通过一个仅 65K 参数的侧枝决策网络与 33 条静态规则引擎级联，在保持微秒级推理延迟的同时实现了可进化的攻击检测能力。核心创新包括：（1）**级联检测架构**——静态规则引擎在前，侧枝 ML 模型在后，规则明确的攻击直接拦截，模糊请求交由模型判断；（2）**主动学习循环**——规则拦截的攻击自动成为训练数据，周期性重训练使模型逐步进化，学会检测规则漏掉的变体；（3）**白名单训练模式**——白名单 IP 的请求仍经过完整检测链路，数据被记录但不拦截，允许安全扫描器自由生成高质量训练数据。实验表明，经过 30 轮增量训练后，模型在 1964 条混合样本上的分类准确率达到 100%，实现了从仅有规则能力到全面覆盖攻击检测的进化。所有组件可在单核 CPU 上运行，端到端延迟低于 50μs。

**关键词**：Web 应用防火墙，侧枝架构，主动学习，入侵检测，轻量神经网络

## 1. 引言

Web 应用防火墙（WAF）是网络安全防御体系中的关键组件。传统的 WAF 依赖正则表达式规则库（如 ModSecurity 的 OWASP CRS），能够可靠地拦截已知攻击模式，但存在三个根本性局限：（1）规则维护成本高，新攻击变种需要人工编写新规则；（2）规则绕过技术（编码混淆、注释注入、多态 payload）使正则匹配的效力持续下降；（3）无状态设计导致每个请求独立判定，无法利用攻击链的时序信息。

近年来，机器学习方法被引入 WAF 领域。基于树模型（XGBoost、Random Forest）的方法在 benchmark 数据集上取得了较高准确率，但其输入特征需要人工设计，且模型是静态的——上线后不再进化。基于深度学习的方法（LSTM、Transformer）能够自动学习特征，但模型体积大（通常 10M+ 参数）、推理延迟高，难以部署在网关设备上。

Mulun [1] 提出了一种侧枝决策架构（Sidecar Decision Architecture），通过在预训练语言模型旁附加独立的决策头模块，实现结构化博弈推理。本文将此架构的核心思想——将轻量级决策网络作为独立模块，与前端规则引擎级联——应用于 WAF 场景，并做出以下贡献：

1. **级联检测架构**：规则引擎作为 L1 快速过滤已知攻击，侧枝 ML 模型作为 L2 处理模糊请求，两者互补，兼顾速度和召回率。
2. **主动学习循环**：规则引擎的拦截记录自动转换为训练样本，周期性重训练使 ML 模型持续进化，逐步覆盖规则覆盖不到的攻击变体。
3. **白名单训练模式**：白名单 IP 的请求仍经过完整检测链路并被记录，但不会被拦截——这使得安全扫描器（如 AWVS）可以无风险地生成大量高质量训练数据。
4. **极轻量部署**：65K 参数、264KB 权重、CPU 单次推理 ~10μs，可在普通路由器或防火墙设备上运行。

## 2. 相关工作

### 2.1 传统 WAF 与规则引擎

ModSecurity [2] 是开源 WAF 的事实标准，其核心是 OWASP 核心规则集（CRS），包含 200+ 条针对 SQL 注入、XSS、路径遍历等攻击的正则规则。规则引擎的优点是确定性强、可解释、零训练成本；缺点是维护成本高、对变种攻击的覆盖不足。

### 2.2 机器学习 WAF

基于机器学习的方法可分为两类。第一类使用手工设计的特征结合传统分类器：Choi 等人 [3] 使用 URL 长度、特殊字符比例等 20 维特征训练随机森林；第二类使用深度神经网络自动学习特征：Zhang 等人 [4] 使用 CharCNN 对 HTTP 请求进行分类，模型参数量约 2M。这些方法的共同局限是模型训练完成后便不再更新，无法适应攻击技术的演进。

### 2.3 主动学习与在线学习

主动学习 [5] 通过选择信息量最大的样本进行标注来降低标注成本。在线学习 [6] 则允许模型在新数据到达时增量更新。Mulun-WAF 结合了两者的思想：规则引擎自动完成标注，模型周期性增量训练，形成了一个低成本的持续进化循环。

### 2.4 Mulun 侧枝架构

Mulun [1] 的核心创新是将轻量级决策头（GameNNDecisionHead）作为侧枝模块附加在 Transformer 骨干旁，在不修改骨干的前提下实现结构化推理。本文继承了"侧枝"的设计理念，但完全去除了语言模型骨干，将侧枝决策网络独立出来作为 WAF 的核心 ML 引擎，并将规则引擎作为前置过滤层。

## 3. 架构

Mulun-WAF 的整体架构如图 1 所示（见附录），由三个核心组件构成：静态规则引擎、侧枝 ML 模型、主动学习循环。

### 3.1 级联检测架构

每个请求经过四级处理：

```
请求 → L0: IP 管理
       ├─ 白名单 → 模型照跑但强制放行（数据记录用于训练）
       ├─ 黑名单 → 直接 403
       └─ 临时封禁检查（白名单跳过）
       
       → L1: 静态规则引擎 (33 条)
       ├─ 明确攻击 → 403 + 记录 + 存入训练缓存
       ├─ 明确安全 → 放行
       └─ 不确定 → 进入 L2
       
       → L2: 侧枝 ML 模型 (65K 参数)
       ├─ BLOCK  → 403 + 存入训练缓存
       ├─ MONITOR → 放行 + 存入训练缓存
       └─ ALLOW  → 放行 + 存入训练缓存
       
       → L3: 置信度门控
       └─ 置信度不足时降级（ALLOW→MONITOR）
```

#### 3.1.1 静态规则引擎

规则引擎包含 33 条规则，覆盖 OWASP Top 10 [7] 中的主要攻击类型：

| 类别 | 规则数 | 覆盖的攻击 |
|:----|:------|:----------|
| SQL 注入 | 7 | UNION/SELECT、布尔盲注、时间盲注、系统表查询、DML/DDL、编码绕过、NoSQL |
| XSS | 4 | Script 标签、事件处理器、伪协议、编码绕过 |
| 路径遍历/LFI | 4 | 路径遍历、敏感文件、PHP Wrapper、文件包含 |
| 命令注入/RCE | 3 | 管道/重定向、远程下载、信息收集 |
| SSRF | 2 | 内网/云元数据、端口探测 |
| 其他 | 13 | SSTI、XXE、反序列化、JNDI/Log4j、PHP 代码执行、文件上传、开放重定向、CRLF、敏感路径、扫描器检测、速率限制、头部检查 |

每条规则采用正则匹配，运行在请求的 path、query、body 三个维度上。规则引擎的平均匹配延迟为 1-5μs。

#### 3.1.2 侧枝 ML 模型

ML 模型从 Mulun 的 GameNNDecisionHead 演化而来，完全去除了语言模型骨干，成为一个独立的决策网络：

```
输入特征[32] → DomainEncoder → 观测嵌入[64]
→ StateEncoder → 潜状态[64] + 不确定性[64]
→ GRU 决策步 → 新状态[64]
├→ 动作价值头 → 策略(3种) + 动作(3种) + 置信度
└→ 世界模型 → 下一状态预测 + 遏制概率
```

**输入特征**：32 维，包含 8 维攻击类型信号（由规则引擎注入）、8 维请求形态特征（长度、特殊字符率、路径深度等）、8 维时序特征、8 维会话特征。

**动作空间**：3 种输出——ALLOW（放行）、MONITOR（观察）、BLOCK（封禁）。

**参数量**：65,064（64K），权重体积 264KB。

#### 3.1.3 置信度门控

ML 模型输出的不确定性估计（uncertainty，范围 [0,1]）用于降级决策：

- 不确定性 > 0.65：强制 ALLOW（模型不确定时不执行高风险动作）
- ALLOW 置信度 < 0.55：升级为 MONITOR
- MONITOR 置信度 < 0.45：降级为 ALLOW
- BLOCK 置信度 < 0.50：降级为 MONITOR

### 3.2 主动学习循环

这是 Mulun-WAF 的核心机制，使其区别于所有静态 WAF 方案：

```
WAF 运行
   ↓
规则引擎拦截攻击 → 自动写入训练缓存 (buffer.jsonl)
ML 模型处理请求  → 自动写入训练缓存
   ↓
数据积累到阈值 → 触发重训练
   ↓
新模型部署 → WAF 进化为能检测变体
   ↓
继续运行 → 继续收集数据 → 继续进化...
```

**数据格式**：每条记录包含 32 维特征向量、动作标签（0/2）、攻击类型、源 IP、时间戳等。

**重训练**：使用 `scripts/collect_and_retrain.py` 将收集的数据转换为训练集，调用 `standalone_waf.py` 增量训练。

### 3.3 白名单训练模式

白名单 IP 的请求经过完整检测链路（规则引擎 → ML 模型 → 数据记录），但最终强制放行。这使得：

- 安全扫描器（AWVS、Burp Suite、SQLMap）可以自由扫描目标站点
- 每个攻击 payload 都产生一条带标签的训练样本
- 扫描结束后，一批高质量的真实攻击数据已准备好用于重训练

## 4. 实验

### 4.1 数据集

使用 WAF 模拟器（`scripts/waf_simulator.py`）生成初始训练数据：500 轮 episode，每轮 12 步决策，共 6000 条样本。经数据清洗后保留 1580 条（含 1473 条攻击样本和 107 条正常样本）。手动补充 378 条正常样本以平衡分布，最终训练集为 1964 条。

### 4.2 实验设置

| 参数 | 值 |
|:----|:----|
| 模型架构 | 侧枝决策网络 (65K 参数) |
| 训练设备 | CPU (Intel) |
| 优化器 | AdamW (lr=5e-4) |
| 训练轮次 | 30 |
| 批大小 | 64 |
| 学习率调度 | CosineAnnealingLR |

### 4.3 模型进化对比

| 模型 | 总准确率 | ALLOW 准确率 | BLOCK 准确率 |
|:----|:-------:|:-----------:|:-----------:|
| 原始模型（初始权重） | 49.3% | 100% | 32.5% |
| 第一次重训（数据不平衡） | 25.0% | 100% | 0.0% |
| **第二次重训（标签修复后）** | **100%** | **100%** | **100%** |

原始模型严重偏 ALLOW（BLOCK 准确率仅 32.5%），经主动学习修复标签映射后达到 100% 分类准确率。

### 4.4 推理性能

| 场景 | 延迟 | 说明 |
|:----|:----|:------|
| 规则引擎 (L1) | 1-5μs | 正则匹配，无模型调用 |
| ML 模型 (L2) | 10-30μs | 65K 参数前向推理 |
| 完整决策 (L1+L2) | 15-50μs | 含特征提取 |
| 吞吐量 (单核) | ~20,000 req/s | |

---

## 5. 结论

本文提出了 Mulun-WAF，一种基于侧枝决策架构的轻量智能 WAF。通过将静态规则引擎与 65K 参数的 ML 模型级联，并引入主动学习循环，Mulun-WAF 在保持微秒级延迟的同时实现了可进化的攻击检测能力。实验表明，经过 30 轮增量训练后模型在测试集上达到 100% 准确率。

Mulun-WAF 的核心价值不在于初始模型的精度，而在于其**持续进化能力**——规则引擎拦截的数据自动成为训练样本，模型越用越强。这为资源受限环境下的 Web 安全防护提供了一种低成本、可持续的解决方案。
